# 渐进式旅行规划

## 先分清三类信息

1. 将稳定的人格与旅行价值取向写入 `.codex-travel/trip-map-builder/MEMORY.md`。
2. 将同行者、日期、地点、交通、住宿方向、身体状态、当前预算、候选和待决策项写入当前旅行的结构化数据。
3. 将智能体应该如何规划、研究、提问和展示的规则只写入本技能。

不得把单次旅行事实、临时身体状况、项目路径、当前候选或工作流程写入 `MEMORY.md`。更新记忆前，先问：“换一次完全不同的旅行，这条仍能描述用户本人吗？”不能则不要写入。

## 按顺序工作

1. 读取当前旅行的原始资料、结构化数据和人格记忆，区分已确认事实、推导、偏好与待决策项。
2. 先设计旅行内容与每日路线，再由路线反推住宿区域；不要先搜酒店再硬凑行程。
3. 只在方向足够明确后搜索真实住宿，使用准确入住日期、退房日期、人数和房间数。
4. 方向确认后提供超过3个有真实差异的候选；卡片保持紧凑，便于横向比较。
5. 每轮只提出一个真正影响下一步的决策，并基于已经搜到的真实结果提问；不要空问偏好。
6. 不要过早生成庞大、固定、看似最终的全程方案。偏好逐步收敛后，再形成2—3套带小选项和替代方案的完整路线。
7. 每次取得新回答后，同步更新结构化数据、研究记录和用户看板，保留被淘汰候选及停用原因。

## 维护双层产物

- 为智能体维护结构化原始数据，保存事实、来源、核验状态、价格口径、候选状态和决策历史。
- 为旅行者维护可直接打开的静态 HTML 看板，突出当前结论、真实候选、主要取舍与下一步唯一决策。
- 在对话中提问，不在 HTML 看板中制作偏好表单。
- 看板首先服务决策，不堆砌研究过程；卡片写清“适合什么”和“主要取舍”。

## 统一表达与价格口径

- 除地名、酒店或平台品牌、交通缩写、货币代码和必要技术标识外，所有面向人的标题、说明、状态与偏好记录均使用简体中文。
- 订房平台支持时直接选择人民币。只能取得外币报价时，保留原币金额并明确写出换算汇率，不把换算价描述成平台人民币报价。
- 记录搜索日期、准确入住与退房日期、人数、房间数、晚数、币种、税费、登录或会员条件、取消条款、付款时间和直达链接。
- 使用Booking时优先从中国站 `www.booking.cn` 进入和核验；看板链接也优先保留中国站直达链接，只有中国站无法打开或缺少关键字段时才回退其他地区域名，并记录回退原因。
- 不把会员价写成公开价；页面没有显示税费或取消条件时，不自行推断。
- 报价是动态快照。后续重新核验时更新当前价，同时保留必要的历史口径。

## 保持行程有弹性

- 按地理顺路程度组合活动，避免无意义横穿城市。
- 为天气、体力、排队和身体状态准备可交换或可跳过的安排。
- 景点数量服务体验，不以打卡数量作为完成标准。
- 已确定交通不重复规划；截图或表格中的空白时间仍属于规划范围。

## 收口检查

结束一轮前检查：

- 路线是否先于住宿区域决定。
- 精确报价是否匹配日期、人数、房间数与晚数。
- 已确认方向是否拥有超过3个可比较候选。
- 看板是否使用中文主文案和人民币主价格。
- 当前只留下一个需要用户回答的主要问题。
- `MEMORY.md` 是否仅保留稳定人格，没有单次旅行事实或工作方法。
